## Cypress Testing Rules

When generating Cypress tests:

### General Principles

- prefer stable deterministic tests
- avoid flaky timing-based assertions
- avoid unnecessary waits
- test real user behavior
- keep tests isolated and independent

### Selectors

- ALWAYS use `data-cy`
- prefer semantic `data-cy` naming
- avoid CSS class selectors
- avoid deeply nested selectors
- avoid text-based selectors when possible
- do not use autogenerated UI-library selectors

Examples:

- `data-cy="save-button"`
- `data-cy="review-dialog"`
- `data-cy="metric-slider"`
- `data-cy="submit-review"`

### Navigation

- verify URL changes where applicable
- verify browser back/forward behavior
- verify refresh persistence for route-based flows

### Assertions

- assert visible user outcomes
- avoid implementation-detail assertions
- prefer explicit assertions over implicit assumptions

### Timing

- NEVER use `cy.wait(milliseconds)`
- use retryable Cypress assertions instead
- wait for real UI states or network completion

### Network Handling

- use `cy.intercept()` when appropriate
- avoid unstable external dependencies

### Accessibility

- verify keyboard accessibility where relevant
- verify focus behavior for dialogs/forms
- verify ESC handling for modals if supported

### Test Structure

Use:

- Arrange
- Act
- Assert

Keep:

- one responsibility per test
- reusable helpers for repeated flows

### Forbidden

- arbitrary waits
- brittle DOM traversal
- forced clicks unless justified
- testing internal component state directly

### Before Generating Tests

First analyze:

1. feature flow
2. async dependencies
3. loading states
4. existing test utilities
5. required `data-cy` selectors

If required selectors are missing:

- add minimal `data-cy` attributes only
- avoid unrelated component modifications

### After Generating Tests

Explain:

- covered scenarios
- uncovered risks
- potential flaky areas
